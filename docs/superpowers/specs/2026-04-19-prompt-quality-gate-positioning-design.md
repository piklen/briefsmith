# Prompt Quality Gate / Task Brief Compiler 定位说明

## 1. 结论

本项目的真实目标，不应再被理解为“prompt 存储工具”或“prompt 润色器”。

更准确的定义是：

> 一个面向 AI coding 宿主的 Prompt Quality Gate / Task Brief Compiler。

它的职责是在 AI 真正执行前，对用户原始输入做一次 **歧义压缩**：

- 能补的补。
- 不能补的问。
- 能明确边界的就提前明确边界。
- 最终输出给宿主 AI 的不是原始口语，而是更稳定的执行前上下文。

## 2. 它解决的到底是什么问题

### 2.1 不是“句子不够漂亮”

用户说“帮我优化”，问题通常不在语言表面，而在执行信息缺失。

例如这句话里至少可能缺少：

- 优化对象：优化什么
- 成功标准：为什么优化、什么算完成
- 约束：哪些东西不能动
- 输出要求：最后要交付什么
- 验证要求：如何证明这次改动有效

因此，本项目面对的不是语言修辞问题，而是 **任务可执行性问题**。

### 2.2 本质是执行前的信息分流

对模糊请求，系统需要做的不是盲目生成更长 prompt，而是判断：

1. 缺哪些关键槽位。
2. 哪些槽位可以通过历史、profile、项目上下文高置信补全。
3. 哪些槽位如果继续猜，会显著提高误伤风险。
4. 最终应该直接编译，还是先追问用户。

所以本项目的核心能力可以概括为：

`模糊意图 -> 槽位检测 -> 上下文补全 -> 置信度判断 -> 编译或追问`

## 3. 产品边界

### 3.1 要做的事

- 导入和检索本地历史 prompt
- 从历史中提炼用户稳定偏好和任务特征
- 对当前输入做缺槽检测
- 结合历史和 profile 自动补全高置信上下文
- 在低置信或高风险情况下返回最小必要追问
- 按宿主和框架渲染成可消费的 brief

### 3.2 不该被当成的东西

- 不是 GUI 优先产品
- 不是简单 prompt 收藏夹
- 不是把所有输入都改写成长 prompt 的“润色器”
- 不是越权替用户拍板的自动决策器

## 4. 真实产品闭环

当前闭环应该理解为：

`历史 prompt -> 检索复用 -> profile 推断 -> 当前输入补全 -> preflight 决策 -> 宿主 AI 执行`

其中：

- `import / reindex / find / favorites / tags` 负责“记住和找回”
- `profile` 负责“形成稳定偏好”
- `compile / preflight` 负责“把模糊输入变成可执行任务”
- `host adapters` 负责“把结果接进不同宿主”

所以历史能力是支撑层，不是产品终点。

## 5. 什么叫“更优提示词”

从语言与执行的共同视角看，更优提示词不等于更长，而等于更少解释分叉。

一个更优的执行型提示词，通常至少要尽量明确：

- `target`：做什么对象
- `success_criteria`：为什么做，怎样算完成
- `constraints`：什么不能动
- `output_format`：最后给什么形态的结果
- `verification`：如何验证结果
- `non_goals`：这次明确不做什么

如果这些内容无法明确，本项目的价值就不是替用户脑补，而是让 AI 先问对问题。

## 6. 决策原则

### 6.1 什么时候直接编译

满足以下条件时，应该偏向直接编译：

- 任务对象已经足够明确
- 成功标准可从词义或历史模式高置信推断
- 风险边界已经在输入中写明，或是稳定默认值
- 缺失的信息不会显著改变执行方向

### 6.2 什么时候必须追问

出现以下情况时，应该追问而不是继续猜：

- 连任务对象都不清楚
- 成功标准缺失，且不同解释会导致不同实现方向
- 约束未知，可能误改外部行为、接口、数据结构或部署方式
- 输出要求未知，可能导致产物形态偏差
- 验证方式未知，可能无法判断任务是否真的完成

### 6.3 追问的原则

- 只问真正影响结果的问题
- 一次优先解决一个关键缺口
- 问题本身要尽量低歧义
- 不为了“形式完整”而机械追问

## 7. 对当前实现的解释

当前实现已经具备这个方向的基础骨架：

- `importers/*`：提供历史来源
- `profile/profile-service.ts`：提供偏好和任务特征
- `compiler/slot-detector.ts`：识别缺槽
- `compiler/compiler.ts`：做补全与编译
- `cli/commands/preflight.ts`：做 host-aware 决策与 evidence 输出
- `host/*`：把运行时能力接入 Claude / Codex / OpenCode

这说明项目方向本身没有偏，只是产品定义还可以更精确。

## 8. 当前槽位与后续补强方向

### 8.1 当前已落地槽位

- `target`
- `success_criteria`
- `constraints`
- `output_format`

这四个槽位已经足够支撑第一版“模糊输入预处理”。

### 8.2 推荐优先新增的槽位

#### `problem_signal`

用于描述用户当前看到的现象或问题信号。

适用场景：

- 修 bug
- 排障
- 回归分析

价值：

- 区分“要优化”和“哪里真的有问题”
- 避免 AI 只做表面重构

#### `verification`

用于描述完成后如何验证。

价值：

- 把“改完了”变成“证明改对了”
- 对 coding 场景尤为重要

#### `non_goals`

用于描述这次明确不做什么。

价值：

- 降低范围扩散
- 控制 AI 过度发挥

#### `scope`

用于描述改动范围。

价值：

- 区分单文件、小模块、全链路优化
- 帮助宿主 AI 选择合适的推进方式

#### `risk_boundary`

用于描述哪些变更属于高风险，不应默认推进。

价值：

- 对接口、数据库、并发、权限、部署等问题更稳
- 可以和 policy / host threshold 形成更明确联动

## 9. 对未来实现的直接指导

如果后续继续演进，本项目应优先朝下面三个方向收敛：

1. 把 `compile / preflight` 继续做成统一内核，减少 CLI 与 host 侧重复逻辑。
2. 扩展可表达的槽位，但保持“最小必要追问”原则，不把系统做成机械表单。
3. 强化 evidence，使宿主和用户都能理解“为什么这次问了”或“为什么这次直接编译了”。

## 10. 对外一句话描述

可以把本项目对外描述为：

> 一个帮助 AI coding 宿主把模糊用户输入转换成高质量执行 brief 的本地运行时。

如果要更强调行为机制，可以补一句：

> 它会优先利用历史 prompt、用户偏好和项目策略补全上下文；只有在关键信息不足时才触发最小必要追问。
